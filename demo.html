★ The one-sentence pitch

Sessions are never labelled with a persona. The platform watches raw API traffic, then statistically rediscovers guest / customer / admin personas and their behavior flows, generates executable tests from those flows, and runs them against a golden OpenAPI oracle to produce a red/green regression report.

traffic→ logs (ELK)→ session flows→ mined behavior + personas→ Playwright specs→ run vs golden→ report

Two claims worth demoing

1. Emergent persona discovery — personas fall out of mined endpoint sequences, then are scored against held-out JWT user_role ground truth (precision/recall per persona).

2. Holdout validation — the registered-customer register → login → checkout sequence exists only in LLM-varied traffic. The engine is expected to rediscover it; a negative control proves it doesn't hallucinate flows that were never injected.

Where the AI is — and isn't

LLM (Haiku 4.5 / Sonnet 4.6): generates LLM-varied traffic, names flows, flags anomalies, recommends assertions, and triages failures.

Deterministic (no key needed): mining, persona classification, the golden oracle, and the pass/fail gate. The LLM is never on the oracle or gate path — so the regression demo stays byte-stable.

A Demo path A — drive everything from the dashboard (recommended for a live audience)

The dashboard is the showpiece: the whole authoring pipeline is wired to buttons, so you never have to touch a terminal in front of the room. Open localhost:5173.

  1. ① Generate traffic & ingest

    Kick off synthetic traffic against Medusa, then ingest the resulting logs from Elasticsearch into session flows + golden candidates. Show the live log count climbing in Kibana while this runs.

  2. ② Mine behavior + personas

    Run the behavior engine. Point out the validation report: per-persona precision/recall vs. the held-out JWT role, the holdout recovery support count where it rediscovers the customer checkout, and the negative control. This is the "AI-driven, not scripted" proof.

  3. ③ Generate tests

    Generate Playwright API specs from the mined candidates — emitted per persona under generated-tests/{guest,customer,admin}. Browse the generated specs in the dashboard.

  4. ④ Run the suite & read the report

    Run the suite from the browser and watch the report render green with per-persona / per-flow / per-endpoint attribution. Optionally trigger triage for advisory root-cause chips, and the repair agent for any wrong-state resolvers.
The dashboard drives /api/pipeline/run (mine / generate / repair / triage), /api/tests/run, and /api/tests/delete — all single-flight and allowlisted. No CLI needed. The suite is API-only Playwright (request fixture, no browsers).

B Demo path B — the CLI pipeline (scripted, reproducible)

Same pipeline, one command per stage. Good for a recorded walkthrough or when you want to narrate each step.

# 3. Generate synthetic traffic against Medusa
npm run traffic:generate

# 4. Ingest ES logs -> session flows + golden candidates
npm run ingest:run

# 5. Mine behavior flows + emergent personas (writes candidates + validation report)
npm run behavior:mine

# 6. Generate Playwright API tests from the candidates
npm run script-generator:generate

# 7. Execute the generated suite (writes reports/report.{json,html})
npm run test:all

# 8. (Optional) Advisory triage of any failures -> reports/triage.json
npm run triage

# 9. Open the report
open reports/report.html

Scope the run with npm run test:guest / test:customer / test:admin / test:happy / test:failure.

C Demo path C — catch a regression live (the money shot 🎯)

This is the most compelling part to show. You break the backend with a feature toggle, watch the suite go red with full attribution, then revert and watch it return to green. 

# Restart Medusa with the regression toggle on
REGRESSION_DEMO=carts_complete_500 docker-compose up -d medusa

# Re-run the customer suite — the checkout /complete now 500s
npm run test:customer
open reports/report.html        # RED, with persona/flow/endpoint attribution

# Revert: unset the toggle and restart
docker-compose up -d medusa     # (without REGRESSION_DEMO)
npm run test:customer           # GREEN again — byte-stable
Why it lands: the failing test is a generated customer-checkout spec that reaches /store/carts/{id}/complete — it wasn't hand-written for this bug. The oracle and gate are deterministic, so the same toggle produces the same red report every time.
Note: some residual customer/guest failures are expected — the SUT gates all cart/checkout mutations behind an authenticated customer, so guest-cart candidates are intentionally unsatisfiable. That's documented drift, not a platform bug.

3 Talking points to hit

Emergent, not scripted

Personas are derived from endpoint sequences + statuses, then scored against held-out JWT roles. Show the precision/recall numbers.

Holdout + negative control

It rediscovers the customer checkout that lived only in LLM-varied traffic — and provably does not report flows that were never injected.

Deterministic gate

The golden OpenAPI oracle and pass/fail gate never touch the LLM, so the regression demo is reproducible to the byte.

5 Troubleshooting

Dashboard shows zeros

A stale container can drop new compose mounts. Force-recreate it:

docker-compose up -d --force-recreate platform-dashboard

Regenerating the full suite

A full regen requires clearing generated-tests/ first (the miner won't overwrite an existing suite in place).

Verify offline (no live stack)

Every service ships a fixture-backed check. Run the whole sign-off chain:

npm run check:phase14

Reset everything

Tear down with volumes if state gets weird, then bring it back up.

npm run stack:reset
npm run compose:up
