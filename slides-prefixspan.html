Why PrefixSpan?

Choosing a sequential pattern mining algorithm to recover user journeys from API traffic — and why not GSP, SPADE, or SPAM.

The task

What we're actually mining

Input

API traffic grouped into sessions → canonical token sequences (METHOD endpoint), consecutive dups collapsed.

Goal

Recover variable-length frequent sub-journeys (order-preserving, gaps allowed) — e.g. the registered-customer checkout backbone, despite browsing noise interleaved around it.

The requirement is not raw throughput. It is constrained, closed, fairness-budgeted, deterministic mining. That framing decides the algorithm.
Context

We don't run "textbook" PrefixSpan

Our miner is a hardened variant. Each extension is native to pattern-growth — and this is the whole argument.

Closed

Emit only closed patterns — drop the sub-pattern explosion.

Gap-bounded

maxGap collapses spurious long-distance patterns.

Fair per-root

Each starting item gets its own budget — rare personas survive.

Deterministic

Pinned ordering: same input → same output, always.

The candidates

Four algorithms, one fit

AlgorithmParadigmWhy it's wrong for usVerdict
GSP Apriori / breadth-first Candidate-generate-and-test + one DB scan per length. Shared checkout backbone → candidate blow-up. dominated
SPADE Vertical (id-lists) Id-lists blow up when high-support tokens co-occur — exactly our case. Constraints don't graft on cleanly. memory + rigid
SPAM Vertical (bitmaps) Fastest on dense/long sequences, but bitmaps sized to total DB length — wasteful for sparse sessions. wrong tradeoff
PrefixSpan Pattern-growth Projected databases, no candidate generation — and the projection IS our extension point. ✓ fit

Why not the vertical algorithms

Speed we don't need, cost we can't avoid

SPADE / SPAM strengths

  • Fewer DB scans; bitwise/id-list joins
  • Win on dense, long-sequence benchmarks

Our reality

  • Sparse, moderate-N session traffic
  • minSupport = 3, gap-bounded, closed → mining is not the bottleneck
  • Projected pointers stay lean; bitmaps/id-lists scale with total DB size regardless of sparsity
Choosing SPAM optimizes a dimension that doesn't bind (speed) at the cost of the one that does (extensibility).
Intellectual honesty

The claim we don't make

PrefixSpan is not universally fastest — SPAM typically beats it on dense, long-sequence benchmarks.

Our defense is fit and extensibility, not benchmark speed. That is a stronger and more truthful position — and it's the one our requirements actually demand.

Takeaway

We chose PrefixSpan because our requirements are constrained, closed, fairness-budgeted, and deterministic mining — and the pattern-growth projected-database model hosts all four natively, whereas GSP is dominated by candidate blow-up and the vertical algorithms trade memory for a speed advantage our data never needs.
