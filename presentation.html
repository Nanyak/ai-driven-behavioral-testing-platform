AI-Driven Behavioral Testing Platform

It drives synthetic traffic at a real Medusa e-commerce backend, captures every request as a structured log, mines the raw log stream to discover how users actually behave, regenerates that behavior as executable contract-checked tests, and runs them to catch regressions.

1 · The problem & the claim

What makes it “AI-driven,” not a scripted round-trip

The hard part of regression testing isn't running tests — it's knowing which behaviors are worth testing and keeping them faithful to real usage. Hand-written suites drift from reality. This platform derives the suite from observed behavior, and makes two falsifiable claims to prove it isn't cheating:

① Emergent persona discovery

Sessions are not labeled with a persona. Guest / customer / admin personas fall out of the mined endpoint sequences and their response statuses — then we score the derived persona against held-out JWT user_role ground truth (precision/recall per persona).

The classifier reads endpoint + status only. The ground-truth label is read in exactly one place — the validation module, after classification — so the accuracy number is a real measurement, not a self-fulfilling label.

② Holdout validation

The registered-customer register → login → checkout backbone exists only in LLM-varied traffic — it is never scripted. The mining engine is expected to re-discover it from statistical co-occurrence, reported as a support count.

A negative control proves the engine doesn't hallucinate: an un-injected flow (a successful POST /store/returns) must surface with support 0.

SAY THIS  “If the engine were just replaying a script, the holdout couldn't pass and the negative control couldn't stay at zero. The fact that both hold is the evidence that it's mining behavior, not echoing it.”

4 · Implementation · the mining engine (from scratch)

How the mining engine actually works

This is the heart of the system, so we'll build it up from zero — no prior knowledge assumed. First: what the data looks like. Then each of the three algorithms, with a worked example on real e-commerce traffic. Then why these algorithms and not ML.

Step 0 · From raw logs to a “sentence” of steps

Before any mining, every user session is reduced to an ordered list of tokens. A token is just METHOD endpoint — the URL is already normalized by ingestion, so a real cart id like cart_01H… becomes the placeholder {id}. Think of each session as a sentence, and each token as a word.

guest browsecartautherror
Session A
GET /store/products› GET /store/products/{id}› GET /store/products› POST /auth/.../register› POST /store/carts› POST /store/carts/{id}/line-items› POST /store/carts/{id}/complete

Two cleanups happen first (signature/signature.ts → canonicalTokens): (1) consecutive identical tokens collapse to one (a 200-then-304 re-fetch of the same URL is one habit, not two); (2) tokens are interned — each distinct string is swapped for an integer id, so the algorithms compare cheap integers, not strings. That's it. The miners never see HTTP, JSON, or status codes — just sequences of integers.

The one word you need: “support.” The support of a pattern = how many distinct sessions contain it. Not how many times it occurs (a session that loops the catalog 50× still counts once). Support is the single measure of “is this behavior real or a fluke,” and all three algorithms are built around it. The threshold is an absolute floor: minSupport = 3 sessions — never “3% of N.”

1

n-gram — the simple baseline

The idea: slide a fixed-size window of n adjacent tokens across each sentence and count how many distinct sessions each window appears in. We use n = 2, 3, 4. It only ever sees tokens that are literally next to each other — no gaps allowed. This captures local habits: “the step right after viewing a product is adding it to the cart.”

Worked example (n = 2, sliding window of 2)

GET /productsGET /products/{id} GET /productsPOST /carts
GET /productsGET /products/{id} GET /productsPOST /carts

Window 1 = (GET /products, GET /products/{id}), window 2 = (GET /products/{id}, GET /products), and so on. Each window the session contains is recorded once for that session. Across all sessions, windows that clear support ≥ 3 survive.

Why we keep it

Dead simple, fast, impossible to get wrong, and explainable — a 2-gram is its own explanation. It's the honest baseline we measure PrefixSpan against (same session-support unit, so it's apples-to-apples).

Why it's not enough

n is fixed, and it demands adjacency. A 7-step checkout never appears as a single 4-gram, and any browsing step wedged into the middle of the journey breaks the window. It can't see the whole story.

mining/ngram.ts — ~40 lines. Its stated job in the code: “be a demo contrast against PrefixSpan, not the primary generator.”

2

PrefixSpan — the primary engine

The problem it solves: a real journey is a subsequence, not a substring. The checkout backbone register → cart → line-items → complete almost never appears with those four steps glued together — there's browsing noise interleaved. We need to find ordered patterns that allow gaps.

subsequence (what PrefixSpan finds): the four steps appear in order, with anything allowed between them.
registerbrowseview cartbrowseline-itemscomplete
substring (what n-gram needs): the four steps glued together with nothing between — rarely happens in real traffic.

How it grows patterns — “projected databases”

PrefixSpan never enumerates all possible patterns (that would explode). Instead it grows one pattern at a time and, for each, keeps a tiny bookmark list called a projection. Walk through it:

  1. Find frequent single steps. Scan every session; any token appearing in ≥ 3 sessions is a starting “root” (e.g. POST /store/carts).
  2. Project. For the root, record one bookmark per supporting session: “in session 7, I matched carts at position 4 — the next step must come from position 5 onward.” That bookmark list is the projected database. Its length = the pattern's support, by construction (one bookmark per session).
  3. Extend. Look just past every bookmark for the next frequent token. If line-items follows in ≥ 3 of those sessions, grow the pattern to carts → line-items and move each bookmark forward.
  4. Recurse until no extension clears support or the pattern hits maxPatternLength (12). Support can only shrink as you extend — which is what makes it terminate.

This is why PrefixSpan is efficient: each step only re-scans the shrinking slice of sessions that still match, not the whole corpus. (In the code: extend() builds the next projection map; grow() recurses; Projection = {seqIdx, start} is literally the bookmark.)

Three engineering decisions that make it work on real traffic

Bounded gap (maxGap = 4)

Allowing unlimited gaps manufactures spurious “patterns” between unrelated steps 50 apart. Real journey steps cluster, so we cap the gap at 4 intervening steps. Gap 0 would just be n-gram; a few steps recovers genuine journeys without the combinatorial blow-up.

Closed patterns only

If carts→line-items and carts→line-items→complete have the same support, the shorter one is redundant — it's just the longer one truncated. We emit only closed patterns (no extension keeps the same support), which collapses a flood of redundant sub-patterns.

Per-root fairness

A global budget lets the highest-volume root (GET /store/products) burn the whole compute budget before low-volume admin reversals are ever reached — starving a persona. So each starting step gets its own budget (perRootCap). Guest, customer, admin all get explored.

This is the algorithm that earns the holdout claim. The register → cart → line-items → complete backbone exists only in unscripted LLM traffic, buried in browsing noise. n-gram can't see it (not adjacent); PrefixSpan recovers it at support 6 precisely because it allows gaps and uses an absolute floor instead of a percentage. (mining/prefixspan.ts)
3

Markov — the “is this transition normal?” signal

The idea: a first-order Markov model answers one question — given you just did step X, how likely is step Y next? We count every adjacent pair (from → to) across all sessions, then P(to | from) = count(from→to) / total transitions out of from.

Worked example

Suppose out of 200 transitions leaving GET /store/products: 150 go to GET /store/products/{id}, 40 to POST /store/carts, and 10 to POST /admin/products.

From → ToCountProbabilityRead as
products → products/{id}1500.75the normal next step
products → carts400.20common
products → admin/products100.05unusual — worth a flag

Any adjacent transition below threshold = 0.02 is surfaced as a rare transition — an anomaly hint. Crucially, this is only fed to the LLM naming/anomaly step as context. Markov never classifies a persona and never decides pass/fail. It's a supporting signal, nothing more. (mining/markov.ts)

How the three fit together

n-gramlocal habits · baseline contrast
PrefixSpanfull journeys · primary candidate source
Markovtransition likelihood · anomaly hints

PrefixSpan produces the candidate flows that become tests. n-gram is the honest baseline that proves PrefixSpan is finding more than adjacency. Markov flags the weird transitions for the LLM to comment on. None of them ever reads the JWT role label — that's the guardrail that keeps persona accuracy a real measurement.

Why it's deterministic (and why that matters)

Every tie is broken by a fixed rule, so the same logs always yield byte-identical output: patterns emit in support desc → length desc → lexicographic by token; roots and extensions are visited in ascending integer id. If mining were random, the 97.3% accuracy and the “recovered at support 6” numbers couldn't be reproduced — and the whole “measured AI claim” would collapse. Determinism isn't a nicety here; it's what makes the validation report evidence.

4b · Implementation · algorithm trade-offs

Why these algorithms — and not deep ML?

This is the question every reviewer asks. The honest framing: we deliberately chose the boring, classical algorithms, and that choice is a feature, not a compromise. Here's the reasoning against each alternative we considered.

ApproachWhat it would addWhy we didn't use it (here)
RNN / LSTM / Transformer
neural sequence model		 Learns long-range dependencies; could rank flows by predicted business value. Non-deterministic & opaque. Two training runs give different weights → the holdout-recovery and persona numbers stop being reproducible. Needs a large labeled corpus we don't have (our point is to work without labels). A wrong test it generates can't be audited line-by-line. It would undermine the exact claim the project makes.
Hidden Markov Model
latent-state sequence model		 Models hidden “intent” states behind the observed endpoints. EM training is sensitive to initialization (local optima → non-reproducible), and you must pre-specify the number of hidden states — which is exactly the persona count we're trying to discover emergently, not assume.
K-means / clustering
on feature vectors		 Could discover finer sub-personas without a fixed taxonomy. Throws away order — a bag-of-endpoints loses “register before checkout,” which is the whole signal. Random initialization + you must pick k up front. (It's listed as genuine future work for sub-persona discovery — as an addition, not a replacement.)
Apriori / association rules
frequent itemsets		 Classic “customers who did X also did Y.” Order-blind, like clustering — finds co-occurrence, not sequence. PrefixSpan is the sequential generalization that keeps order, which is what we need.
Process mining
(e.g. Alpha/Heuristics miner)		 Builds a full process graph / Petri net of the workflow. Aimed at visualizing one global process model; we want ranked, testable, per-persona flow candidates. Heavier dependency for a different output shape. A reasonable future direction, not a fit for “emit specs.”

What classical mining buys us

  • Determinism — same logs, same output, every time. The validation numbers are reproducible.
  • Auditability — every candidate traces to specific supporting sessions; you can read why it exists.
  • No labels, no training — works on raw logs the day you point it at them; the “emergent” claim stays honest.
  • Cheap & offline — runs in seconds on a laptop, needs no GPU, no API key.
  • Explainable to a reviewer — “support 6, gaps allowed, absolute floor” is a sentence; a learned embedding is not.

What we give up (honestly)

  • Long-range dependencies beyond maxGap are invisible.
  • No notion of business value — frequency ≠ importance; a rare-but-critical refund flow ranks low.
  • Fixed persona taxonomy — three tiers, not emergent sub-clusters.
  • No semantic generalization — it treats endpoints as opaque tokens; it doesn't “understand” that two endpoints are similar.

The stated future path keeps the deterministic miner as a verifiable baseline and adds a learned model on top for ranking — never on the gate path.

SAY THIS  “We could have reached for a neural sequence model. But the project's value is its reproducibility — the holdout and the negative control only mean something because the miner is deterministic and auditable. A black box that generates tests you can't explain would defeat the purpose. So the ML is deliberately kept where it's safe: generating realistic traffic and naming flows, never deciding what's a regression.”
5 · Implementation · emergent personas

Personas are derived from attributes, never read from a label

Each mined flow carries attributes — the endpoints it touched and the response statuses it saw. Persona is resolved highest-privilege-wins:

is_admin                         -> admin_operator
requires_auth and not is_admin   -> registered_customer
neither                          -> guest_shopper

has_errors is an orthogonal edge-case overlay, never a competing persona. Every persona is tagged persona_source: "emergent_attributes".

The cart signal — a measured delta, not a guess

The classifier is scored under multiple rule variants so each signal's contribution is an isolated, measured delta against the held-out ground truth:

Rule variantWhat it addsMacro F1Accuracy
endpoint_onlybaseline — endpoints touched0.89685.8%
cart_signal+ auth-gated cart mutation0.96995.8%
cart_read_signal+ auth-gated cart read0.98097.3%
HONESTY  The cart signal is sound only while the SUT's requireCustomerAuth gate actually enforces (a guest POST /store/carts must 401). Some guest→customer reclassifications are ground-truth gaps, not classifier errors: a returning customer reusing a live JWT emits no /auth endpoint, so the JWT label under-counts them. We footnote this rather than hide it.
6 · Implementation · the golden oracle

The OpenAPI contract is the oracle — not recorded bodies

ADR 0001. Logged response values are volatile; the OpenAPI contract is stable. So the assertion oracle resolves each operation/status against the augmented Store/Admin OpenAPI spec, and tightens under-specified spec leaves with observed shapes — but spec-declared fields are never removed by observation.

Shape / type layer

Field existence + JSON types, per operation + status. A shared, auditable ignore-fields list strips volatile fields (id, timestamps, tokens, cart_id, …) so they're never flagged as regressions.

Tier A value rules (implemented)

enum / const / minimum / maximum / format constraints are lifted straight from the spec into each golden's value_rules and gated alongside the shape diff. The spec authors them — promotion is judgement-free.

Where the spec has no entry, it falls back to an observed schema (schema_source: "observed"). oneOf alternatives are skipped (unsound to assert); rules never fire on absent/null values.

SAY THIS  “The oracle is as strict as the contract allows and no stricter. That's deliberate: a test that fails because a UUID changed isn't catching a regression, it's catching noise.”

10 · Honesty · trade-offs

Every design choice, and what it costs

These are the questions a sharp reviewer will ask. Lead with the choice and the cost — it reads as deliberate, not defensive.

Choice · classical mining

n-gram / PrefixSpan / Markov over endpoint tokens. Deterministic, auditable, reproducible.

Cost · no deep ML

Misses long-range dependencies a transformer might catch. Mitigation: determinism is exactly what makes the measured numbers trustworthy; a learned model would be added with this miner kept as a verifiable baseline.

Choice · contract is the oracle

Assert OpenAPI shape/type + spec-declared value rules. Stable, judgement-free.

Cost · no exact-value / cross-field

Won't catch “total ≠ sum of lines” yet. Mitigation: Tier A (spec value rules) shipped; Tier B/C (observed enums, cross-field invariants) are LLM-proposed + HITL-approved, then promoted into the deterministic oracle.

Choice · LLM as enrichment only

Names, flags, suggests. Never decides pass/fail.

Cost · no LLM “judgment” on the gate

Can't use the LLM's reasoning to settle ambiguous failures automatically. Mitigation: that's the point — determinism on the gate path (ADR 0005); the LLM proposes, a human promotes.

Choice · synthetic traffic

Scripted + LLM-varied narratives, with a holdout backbone never scripted.

Cost · not real production data

Behavior distribution is manufactured. Mitigation: holdout + negative control make the methodology defensible; the session-flow contract is source-agnostic — point ingestion at a real ES index and nothing downstream changes.

Choice · API-level Playwright specs

Assert against the API contract directly.

Cost · no browser / UI E2E

Won't catch a broken button. By design: the assertion target is the API contract; UI E2E is a different oracle.

Choice · fixed 3-tier persona taxonomy

guest / customer / admin, mapping to Medusa's real auth tiers.

Cost · no finer sub-personas

No “promo-driven shopper” cluster. Mitigation: emergent from attributes (not labeled); unsupervised clustering of attribute vectors is the future direction.

Choice · file-decoupled stages

Every stage reads/writes JSON on disk.

Cost · not a streaming system

Batch, re-runnable — not real-time. Mitigation: exactly what makes each stage independently inspectable and offline-checkable; ELK is single-node, laptop-bound (sufficient for the demo corpus).
11 · Honesty · scope

Explicitly out of scope

UI / browser E2E

Generated tests are API-level by design; the assertion target is the contract.

Load / performance

Tests behavioral correctness, not throughput or latency SLOs.

Auto-remediation

Detects and attributes regressions; fixing is left to engineers.

The honest summary line: The platform makes a narrow, measurable claim — mine real behavioral sequences, classify personas emergently, regenerate them as contract-checked tests, and catch regressions — and backs each part with a deterministic, reproducible artifact plus a validation report (precision/recall, holdout recovery, negative control). The limitations are scoping choices in service of that defensibility, each with a concrete path to extend.
12 · Q&A prep

Likely questions, crisp answers

“Isn't this just replaying logged requests?”

No — it mines frequent subsequences across many sessions and re-synthesizes a clean spec, with the contract as oracle. The holdout proves it re-discovers behavior it never saw scripted; the negative control proves it doesn't invent flows.

“Where's the AI? It sounds like classical algorithms.”

Two layers. The discovery layer is deliberately classical and deterministic — that's what makes the numbers reproducible. The LLM layer generates realistic unscripted traffic (Haiku) and enriches mined flows with names/anomaly flags/assertion suggestions (Sonnet→Opus), but is kept off the gate path on purpose.

“How do you know the persona accuracy is real and not circular?”

The classifier reads endpoint + status only. The JWT user_role ground truth is read in exactly one place — the validation module, after classification. 97.3% accuracy / 0.98 macro F1 over 331 sessions, admin at perfect precision/recall.

“What happens with no API key?”

Mining and classification run unchanged — they need no key. Only LLM-varied traffic and flow naming degrade; naming falls back to a deterministic offline path. Nothing on the gate path depends on the LLM.

“Would this work on a different backend?”

The stages are decoupled by file contracts and the oracle is OpenAPI-driven, so the approach generalizes to any logged, OpenAPI-described REST service. A second target backend and GraphQL/gRPC oracle adapters are the stated future work.

“Why should I trust a green report?”

Because we can make it go red on command — the regression demo flips a single endpoint to 500 via a reversible env toggle and the mined customer-checkout test catches it, attributed to persona/flow/endpoint. And every offline check (check:phase0…15) runs the same logic against committed fixtures with no live stack.

Closing line

“We didn't write tests for what we think users do. We mined what they actually did, proved the mining is honest with a holdout and a negative control, regenerated it as contract-checked tests, and showed it catch a real regression — all reproducibly, with the AI kept off the decision path.”
